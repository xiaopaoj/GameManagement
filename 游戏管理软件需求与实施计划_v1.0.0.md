## 1. 项目目标

本项目用于统一管理硬盘中的单机游戏原始压缩文件、可游玩目录和游戏存档。

机械硬盘只保存下载得到的原始 ZIP、RAR 压缩文件；用户准备游玩时，软件将所选原始文件复制到指定的固态游戏盘，完成两级解压、游戏目录识别、启动程序识别和存档恢复。游戏归档时，软件保存游戏目录内变化文件以及已确认的 Windows 系统目录存档，并在存档备份完成后清理可游玩目录，释放固态空间。

软件提供 Windows 图形化界面，所有复制、扫描、解压、Hash、备份和清理任务均需要显示状态、进度和错误信息。

## 2. 项目范围

### 2.1 支持范围

- 仅支持 Windows 系统。
- 支持配置多个游戏盘，每次准备、归档或特殊归档前选择目标游戏盘。
- 支持添加多个原始游戏文件扫描路径。
- 支持 ZIP、RAR 识别与解压。
- ZIP 和 RAR 解压能力随软件提供，不依赖用户另行安装解压软件。
- RAR 只要求解压，不要求创建 RAR 文件。
- 备份文件统一创建为无密码 ZIP。
- 支持多层目录、两级压缩、错误扩展名和分卷压缩文件。
- 支持一个游戏配置多个版本，但同一时间只允许一个版本处于可游玩状态。
- 支持游戏目录内存档和经用户确认的 Windows 常见系统目录存档。
- 支持正常退出快照与异常退出快照。
- 支持每日计划备份和手动备份。

### 2.2 暂不支持

- 非 Windows 系统。
- Steam、Epic 等游戏平台安装游戏管理。
- 模拟器游戏专项管理。
- 一个游戏配置多个主启动程序。
- 同一个游戏的多个版本同时保持可游玩状态。
- 创建 RAR 备份。
- 多个外部备份目标目录。
- 在软件内自动恢复外部 ZIP 备份。
- 对游戏进程注入 DLL 或修改进程代码。

## 3. 术语与目录定义

### 3.1 术语

| 术语 | 定义 |
|---|---|
| 原始文件 | 机械盘中下载得到的 ZIP、RAR、分卷文件或包含这些文件的目录 |
| 游戏盘 | 用户配置的固态工作盘，用于保存临时文件、可游玩目录和本地存档 |
| 可游玩目录 | 已完成解压并包含有效主游戏 EXE 的目录 |
| 本地存档 | 保存在游戏盘 `GameSave` 目录中的活动存档、快照和清单 |
| 外部备份 | 保存在单独备份目录中的无密码 ZIP 文件 |
| 文件基线 | 首次解压完成后记录的干净游戏目录文件信息 |
| 普通归档 | 保存存档后处理软件正常准备的可游玩目录 |
| 特殊归档 | 对用户已有的混乱游戏目录与干净基准目录进行比较和存档提取 |

### 3.2 游戏盘目录

每个游戏盘使用独立工作根目录：

```text
{游戏盘工作根目录}/
├─ Games/
│  └─ {游戏ID}/
├─ GameTemp/
│  └─ {游戏ID}/{版本ID}/
├─ GameSave/
│  └─ {游戏ID}/
└─ GameSaveTemp/
   └─ {游戏ID}/{版本ID}/
```

游戏 ID 和版本 ID 使用不含中文的随机 UUID。

## 4. 配置管理

### 4.1 扫描路径

- 支持添加、删除、启用和禁用多个扫描路径。
- 所有扫描路径需要持久化保存。
- 支持手动重新扫描。
- 显示路径状态、所属磁盘、剩余空间和最近扫描时间。
- 路径失效时显示错误并提供重新定位功能。

当前实现支持在扫描路径列表中查看有效状态、所属磁盘、剩余空间和最近扫描时间，并通过独立按钮启用、禁用或重新定位路径。

### 4.2 游戏盘

每个游戏盘配置包含：

- 显示名称。
- 工作根目录。
- 是否启用。
- 是否默认选中。
- 最低剩余空间阈值。
- 磁盘容量、剩余空间和磁盘类型。

执行准备、再次准备、存档复制或特殊归档前必须选择游戏盘。界面显示预计所需空间和操作后预计剩余空间。

当前实现支持配置多个游戏盘、默认游戏盘、启用状态和最低保留空间。设置页面显示路径状态、磁盘类型、总容量、剩余空间和最低保留空间；准备游玩时默认游戏盘排在选择列表首位，但仍由用户最终确认。

### 4.3 外部备份目录

- 当前版本只支持一个外部备份目录。
- 记录卷标、卷序列号、最近盘符和相对目录，降低盘符变化影响。
- 备份盘离线时将任务标记为“等待目标磁盘”。
- 目标磁盘重新上线后自动补执行等待中的计划备份。

## 5. 文件扫描与游戏添加

### 5.1 文件扫描列表

扫描列表显示：

- 文件或目录名称。
- 完整路径。
- 文件类型。
- 文件大小。
- 修改时间。
- 所属磁盘。
- 是否疑似压缩文件。
- 来源形式：直接 ZIP/RAR 压缩文件，或内部包含压缩文件的目录。
- 目录来源的压缩包状态；文件扫描阶段显示“准备时扫描”，不在此阶段递归读取目录内容。
- 是否已经加入游戏库。
- 路径是否有效。

支持按名称、路径、类型、磁盘和添加状态过滤，并支持隐藏已加入游戏库的文件夹。

为避免机械硬盘扫描时因递归读取大量文件及压缩包头尾而长时间卡顿，游戏库文件扫描采用以下边界：

- 只枚举每个已启用扫描路径的直属文件夹。
- 只枚举每个已启用扫描路径直属且扩展名为 `.zip`、`.rar` 的文件。
- 不在游戏库扫描阶段递归进入候选文件夹，也不读取其他直属文件的文件头或尾部。
- 候选文件夹内部的 ZIP、RAR、错误扩展名压缩文件及混淆文件，延迟到“准备游玩”阶段递归识别。
- 扫描过程在后台执行，界面显示已完成数量、总数量、百分比和当前处理路径。
- 扫描完成前禁用主界面其他操作，并禁止关闭主窗口；扫描成功或失败结束后恢复操作。

### 5.2 添加游戏

用户判断某个文件或目录属于游戏后，可将其添加至游戏库：

- 生成随机 UUID 游戏 ID。
- 支持在扫描候选列表中多选并批量添加。
- 添加时不再要求用户输入游戏名称。
- 文件夹候选直接使用文件夹名称作为游戏名称；ZIP、RAR 候选使用去除扩展名后的文件名称作为游戏名称。
- 两种原始来源均可添加：直接选择单个 ZIP/RAR 文件，或选择内部包含一个或多个 ZIP/RAR 的目录。
- 游戏和版本数据需要持久化保存来源形式，后续准备流程不得假定来源一定是目录或一定是文件。
- 不修改实际文件或目录名称。
- 保存原始路径、大小、修改时间和文件指纹。
- 批量添加时自动创建首个游戏版本，版本名称统一为“初始版本”。

### 5.3 游戏版本

一个游戏允许包含多个版本，每个版本独立保存：

- 版本 ID。
- 版本名称和备注；首个批量添加版本默认为“初始版本”，后续新增版本名称由用户填写。
- 原始压缩文件路径及指纹。
- 用户选择的压缩文件或分卷组。
- 第一次与第二次解压规则和密码。
- 游戏根目录识别结果。
- 主游戏 EXE 相对路径。
- 干净文件基线。
- 最后游玩时间。

同一游戏只允许一个版本处于可游玩状态。切换版本前必须完成当前版本归档。

阶段二实现约束：

- 添加游戏或新增版本时，在阻塞式进度窗口中计算来源文件数量、总体积、最后修改时间和 SHA-256 指纹；目录来源使用内部全部文件的相对路径、大小和内容 SHA-256 生成稳定聚合指纹。
- 旧数据缺少来源指纹时，在首次准备游玩前自动补充并持久化。
- 游戏详情支持修改显示名称和备注，备注允许留空，不修改硬盘上的实际名称。
- 版本管理支持添加用户命名版本、编辑版本名称与备注、查看来源状态、将版本设为当前版本、打开原始位置和重新定位来源。
- 当前存在可游玩目录时禁止切换版本；必须先完成后续阶段定义的归档流程。
- 重新定位时重新校验 ZIP/RAR 文件或压缩文件目录，计算新来源元数据，并展示文件名、总体积、修改时间和 SHA-256 的匹配结果，由用户最终确认是否替换。
- 游戏列表、游戏详情和版本列表显示原始路径是否有效。

阶段二当前实施进度：多个扫描路径及持久化、直属候选扫描、名称/路径/类型/磁盘/添加状态筛选、批量添加、游戏名称与备注维护、多版本新增与切换、来源元数据持久化、路径失效显示和人工重新定位已实现。分卷完整性校验按阶段四继续实现。

- 准备游玩。
- 启动游戏。
- 打开游戏目录。
- 归档游戏。
- 特殊归档。
- 手动备份。

## 7. 状态模型

建议使用以下游戏状态：

```text
待配置
未准备
复制中
第一次解压中
第二次解压中
等待确认游戏目录
可游玩
运行中
等待确认系统存档
归档中
已归档
版本切换失败
源文件失效
操作失败
```

归档状态与目录清理状态相互独立。存档保存、校验和外部备份完成后即可标记“已归档”，游戏目录是否成功进入回收站不影响归档状态。

目录状态示例：

```text
目录存在
等待清理
已移入回收站
清理失败
```

## 8. 首次准备流程

### 8.1 准备检查

1. 用户选择游戏版本和目标游戏盘。
2. 检查原始文件路径及格式。
3. 检查同一游戏是否已有其他可游玩版本。
4. 估算复制和解压空间。
5. 检查游戏盘剩余空间。
6. 创建 `GameTemp/{游戏ID}/{版本ID}`。

### 8.2 第一次解压

1. 将机械盘原始文件复制到游戏盘临时目录。来源是单个 ZIP/RAR 时复制该文件；来源是目录时复制该目录及其内部文件。
2. 来源是单个 ZIP/RAR 时直接将该文件作为第一次解压候选；来源是目录时递归扫描其中所有 ZIP、RAR 和分卷组。
3. 存在多个压缩文件或分卷组时，由用户手动选择。
4. 用户输入第一次解压密码，或读取已保存密码。
5. 解压到第一阶段目录并记录结果。

分卷缺失时允许用户确认后尝试解压一次；失败后停止，不自动重试，也不继续后续步骤。

### 8.3 第二层压缩识别

- 递归扫描第一次解压结果。
- 检查文件头部和尾部区域中的 ZIP、RAR 标识，不仅依赖扩展名。需要兼容在 MP4 等伪装数据之后内嵌 ZIP，并在文件末尾保留 ZIP 结束目录的混合文件。
- 对带有 MP4 等前置伪装数据的混合文件，需要定位真实 ZIP/RAR 起始签名，将起始签名之前的伪装前缀排除，生成规范化临时压缩文件后再解压。仅修改扩展名不足以保证内置解压组件能够识别。
- 大文件优先作为候选，但不得仅凭文件大小自动确定。
- 识别出真实格式后，在第一次解压产生的临时目录中将文件改为 `.zip` 或 `.rar`，不修改机械盘原始文件。
- 存在多个候选时由用户手动选择。
- 如果通过文件头部和尾部仍然无法识别压缩文件，选择第一次解压结果中最大的未识别文件，先改名为 `.zip` 尝试带密码解压；失败后再改名为 `.rar` 尝试带密码解压。
- ZIP、RAR 两种尝试均失败时停止准备流程并保留临时目录，不得直接进入 EXE 搜索。

### 8.4 第二次解压

- 第二次密码独立于第一次密码，并与具体压缩文件指纹绑定。
- 解压失败时允许重新输入密码或重新选择文件。
- 成功后进入游戏目录与主 EXE 识别。

### 8.5 游戏目录和主 EXE 识别

从最终解压结果开始按目录层级和目录名称稳定排序进行递归搜索。找到第一个不属于排除名单的有效 EXE 后，将该 EXE 所属文件夹标记为最终游戏目录，并停止继续搜索其他目录。用于确定游戏目录时排除常见辅助程序，例如：

```text
UnityCrashHandler.exe
UnityCrashHandler64.exe
CrashReportClient.exe
UE4PrereqSetup_x64.exe
UEPrereqSetup_x64.exe
unins000.exe
uninstall.exe
setup.exe
vc_redist.x64.exe
DXSETUP.exe
```

确定游戏目录后，列出该目录直属的全部 `.exe` 文件和文件名为 `index.html` 的网页入口，不递归展示子目录文件，由用户选择一个作为游戏启动文件。辅助 EXE 只在“确定游戏目录”阶段被排除，目录确定后仍按“全部 EXE”要求展示。每个游戏版本只保存一个启动文件。其他插件或程序由用户通过“打开游戏目录”自行启动。

识别成功后：

- 提取主 EXE 图标。
- 没有游戏备注时，可以使用目录名称作为候选备注，由用户确认。
- 将最终游戏目录放入 `Games/{游戏ID}`。
- 保存游戏根目录和主 EXE 相对路径。
- 建立干净文件基线。
- 自动永久删除成功任务的 `GameTemp`，无需确认。

当前实施进度：已经实现游戏盘选择、按来源体积四倍估算临时与解压空间、最低剩余空间检查、原始来源复制、两次压缩文件人工选择、ZIP/RAR 头尾区域识别、混合文件真实压缩数据起始位置定位与规范化临时文件生成、错误扩展名临时改名、最大未识别文件依次按 ZIP/RAR 尝试、DPAPI 密码保存、ZIP/RAR 解压、主 EXE 人工确认、最终目录复制和 SHA-256 文件基线。分卷完整性专项检查、压缩内容精确空间估算和 EXE 图标提取仍需继续实现。

## 9. 密码管理

- 第一次和第二次密码分别保存。
- 密码与具体游戏版本及压缩文件指纹绑定。
- 使用 Windows DPAPI 加密存储。
- 默认以掩码显示，用户可以主动查看。
- 支持修改、删除和重新验证。
- 日志不得记录明文密码。
- 外部存档 ZIP 不包含游戏解压密码。

## 10. 文件基线与 Hash

文件基线至少包含：

- 游戏 ID、版本 ID。
- 相对路径。
- 文件大小。
- 修改时间。
- SHA-256。
- 文件类别。

比较时先使用路径、大小和修改时间快速判断，必要时计算 SHA-256。首次准备完成后必须成功保存基线，才能自动清理临时目录。

归档候选默认排除满足以下条件的文件：

- 大于或等于 100 MB。
- 首次基线中已经存在。
- 疑似游戏资源包。
- 未被用户明确确认为存档。

已确认的存档文件不受 100 MB 排除规则限制。小型日志、缓存、截图和配置文件不自动排除，统一交由用户确认。用户排除规则只针对具体文件，不支持目录、通配符或正则表达式。

## 11. 游戏运行监控

### 11.1 进程判断

- 游戏运行与结束完全以主游戏 EXE 状态为准。
- 主 EXE 启动后状态变为“运行中”。
- 主 EXE 退出后本次运行结束。
- 插件、修改器和其他进程不影响运行状态。
- 不进行 DLL 注入，采用进程生命周期监听和文件系统变更通知。

### 11.2 首次系统存档扫描

游戏第一次运行前，对以下常见目录建立快照：

```text
%APPDATA%
%LOCALAPPDATA%
%USERPROFILE%\Documents
%USERPROFILE%\Saved Games
```

主 EXE 退出后比较变化，向用户展示新增、修改和删除的候选文件。用户确认后保存为该游戏的存档目录规则。

### 11.3 后续监控

- 首次确认完成后不再扫描全部常见目录。
- 只监控该游戏已配置的一个或多个存档目录。
- 用户可手动添加、修改、禁用和删除目录。
- 同一个系统目录允许关联多个游戏。
- 共享目录中的文件归属不明确时必须再次由用户确认。

### 11.4 存档候选预览

支持：

- 查看文件路径、大小、修改时间和变化类型。
- 预览文本和常见图片。
- 使用系统默认程序打开文件。
- 在资源管理器中定位文件。
- 确认为存档。
- 排除具体文件。

## 12. 本地存档与快照

建议目录：

```text
GameSave/{游戏ID}/
├─ current/
├─ normal-snapshots/
├─ abnormal-snapshots/
└─ manifest.json
```

- 只有存档发生变化时才创建快照。
- 正常退出快照默认保留三个。
- 异常退出快照单独标记并默认保留三个。
- 异常快照不作为默认恢复首选。
- 主 EXE 异常退出且存档变化时立即创建异常快照。
- 用户允许手动删除快照，删除前确认一次。
- 超过保留数量时不自动删除，标记为建议清理并提醒用户确认。

同一个游戏的不同版本默认共享存档。向不同版本恢复前提示兼容风险，并保留来源版本信息。

## 13. 跨游戏盘存档复制

再次准备时选择了不同游戏盘，应将主存档复制到目标盘：

1. 复制到目标临时目录。
2. 校验临时副本。
3. 目标位置已有存档时，每次询问用户是否覆盖。
4. 用户确认后替换目标存档。
5. 再次校验目标存档。
6. 来源游戏盘存档保留。
7. 将目标副本设置为当前主存档。

该流程属于游戏盘间存档复制，不属于外部归档备份。

## 14. 再次准备与版本切换

机械盘不保存完整可游玩目录，所以再次游玩必须重新解压：

1. 选择目标版本和游戏盘。
2. 检查原始文件，失效时重新定位。
3. 使用保存的压缩文件组、密码和解压规则重新执行两级解压。
4. 使用历史规则识别游戏根目录和主 EXE。
5. 恢复共享存档。
6. 自动清理成功任务临时目录。

切换版本前必须：

1. 结束当前主游戏 EXE。
2. 保存并校验最新存档。
3. 确认最新存档已经完成有效外部备份。
4. 将当前版本标记为已归档。
5. 询问后将当前游戏目录移入回收站。
6. 准备目标版本。

新版本准备失败时只进行状态标记和错误记录，不自动从回收站恢复旧版本。

## 15. 普通归档

归档成功条件：

- 游戏目录内存档收集成功。
- 已配置的系统目录存档收集成功。
- 本地存档快照创建并校验成功。
- 最新存档已经存在有效外部 ZIP 备份。
- 外部 ZIP 校验成功。
- 存档清单写入成功。

如果最新存档尚未完成外部备份：

1. 阻止目录删除。
2. 跳转手动备份页面。
3. 自动选中当前游戏和最新存档。
4. 备份及校验成功后返回归档流程。

满足归档条件后立即将游戏状态标记为“已归档”。随后询问一次是否将游戏目录移入回收站。目录删除成功与否不影响归档状态。

已归档但目录仍存在时允许继续启动游戏；启动后恢复为“可游玩/运行中”，并重新监控存档变化。

## 16. 特殊归档

特殊归档流程：

1. 选择游戏和版本。
2. 选择用户已有的混乱游戏目录。
3. 选择操作游戏盘。
4. 在 `GameSaveTemp` 中从原始压缩文件构建干净基准目录。
5. 与混乱游戏目录进行相对路径、大小、时间和 Hash 比较。
6. 展示新增、修改、缺失及默认排除文件。
7. 用户确认存档候选。
8. 创建并校验本地存档快照。
9. 确认最新存档已经完成有效外部备份。
10. 标记游戏已归档。
11. 询问一次后将混乱游戏目录移入回收站。
12. 成功任务自动永久清理 `GameSaveTemp`。

如果无法建立完整基线，必须标记“无完整基线”，由用户手动选择存档文件，禁止自动判断后直接删除目录。

## 17. 外部备份

### 17.1 备份格式

- 使用无密码 ZIP。
- ZIP 内包含存档文件、`manifest.json`、来源游戏盘标识、游戏和版本信息、原始恢复路径及 Hash 清单。
- 不包含游戏解压密码。

### 17.2 手动备份

用户可以对单个游戏执行手动备份。备份完成后进行 ZIP 可读性和文件 Hash 校验。

### 17.3 每日计划备份

- 只支持每天执行，可配置执行时间。
- 通过 Windows 任务计划程序启动独立的无界面备份组件。
- 即使主程序没有运行，也必须能够执行。
- 对所有游戏盘的完整 `GameSave` 目录进行备份。
- 多游戏盘内容合并到一个 ZIP，并按照游戏盘 ID 分目录保存。
- 整体内容指纹没有变化时不重复生成 ZIP。
- 目标磁盘离线时等待磁盘重新上线后补执行。
- 执行完成、部分失败或等待磁盘时弹出 Windows 通知。
- 完整 `GameSave` 计划备份默认保留三个。
- 超过三个时只提醒，不自动删除；用户确认后删除。

示例结构：

```text
GameSaveBackup_20260719_020000.zip
├─ manifest.json
├─ game-disk-{磁盘ID-1}/GameSave/
└─ game-disk-{磁盘ID-2}/GameSave/
```

## 18. 外部备份查看

软件提供只读查看页面，不提供自动恢复：

- 显示备份路径、大小、SHA-256、时间和校验状态。
- 显示 ZIP 内文件树。
- 显示游戏、版本和原始恢复路径。
- 支持打开备份所在目录。
- 支持使用系统程序打开 ZIP。
- 支持复制恢复路径。
- 支持重新校验。
- 根据 `manifest.json` 显示人工恢复步骤。

人工恢复提示至少包括关闭游戏、备份现有目标文件、将 ZIP 中对应内容复制到游戏目录或 Windows 用户目录，以及由用户决定是否覆盖同名文件。

## 19. 原始文件重新定位与删除

### 19.1 重新定位

- 支持选择新文件或目录。
- 支持按文件名、大小、修改时间和 Hash 匹配。
- 检查 ZIP、RAR 格式和分卷完整性。
- 更新路径前要求用户确认替代版本或匹配结果。

### 19.2 删除原始文件

原始压缩包只能通过独立按钮删除，且需要两次人工确认：

1. 第一次显示游戏、版本、路径、文件数量、总体积和无法重新准备的风险。
2. 第二次要求输入版本名称或固定确认文字。
3. 将主压缩文件及相关分卷移入 Windows 回收站。
4. 不支持回收站时终止操作，不得自动永久删除。

删除游戏版本记录时提示是否同时删除原始文件。选择保留时只删除数据库版本记录；选择删除时完成上述二次确认后再删除版本记录。

## 20. 删除与清理规则

| 对象 | 处理方式 | 确认次数 |
|---|---|---:|
| 成功任务的 `GameTemp` | 永久删除 | 0 |
| 成功任务的 `GameSaveTemp` | 永久删除 | 0 |
| 手动清理失败任务临时目录 | 永久删除 | 1 |
| 普通归档游戏目录 | 回收站 | 1 |
| 特殊归档混乱目录 | 回收站 | 1 |
| 原始压缩包和分卷 | 回收站 | 2 |
| 本地存档快照 | 永久删除 | 1 |
| 外部 ZIP 备份 | 永久删除 | 1 |
| 超额历史快照或备份 | 永久删除 | 1 |
| 游戏版本记录 | 数据库删除 | 1 |
| 游戏主记录 | 数据库删除并输入完整游戏名称 | 1 |

所有无人值守任务不得执行需要人工确认的删除。

删除游戏主记录前必须确认不存在版本、原始文件、可游玩目录、本地存档、外部备份、未完成任务和残留临时目录。所有关联数据清空后，用户仍需输入完整游戏名称并完成最终确认。

记录复制、解压、扫描、Hash、快照、备份和清理任务的进度、当前文件、错误、开始时间和完成时间。

- 文件扫描和来源指纹计算提供取消按钮；准备游玩进度窗口提供取消按钮，并将取消令牌传递到复制、Hash、压缩扫描、解压、游戏目录识别和文件基线步骤。
- 用户取消后状态标记为“已取消”，错误时标记为“失败”，临时目录保留；软件启动时将上次仍为“运行中”的任务标记为“已中断”。
- 任务中心支持查看任务详情、打开临时目录、打开关联游戏，以及在一次人工确认后永久清理失败、取消或中断任务的临时目录。
- 临时目录清理前必须验证目标位于已配置游戏盘的 `GameTemp` 或 `GameSaveTemp` 子目录，禁止删除临时根目录、`Games` 或游戏盘根目录。
- 准备游玩先将最终游戏目录整理到 `GameTemp` 内的暂存目录，文件基线成功后再移动到 `Games/{游戏ID}`，避免失败或取消时留下半成品可游玩目录。
- 成功任务自动永久删除 `GameTemp`，无需确认；清理失败不改变游戏已经准备完成的状态，任务中心保留目录供用户后续手动清理。

## 22. 安全与可靠性要求

- 所有危险删除操作遵守确认规则。
- 原始文件和游戏目录优先进入回收站，不自动降级为永久删除。
- 密码使用 DPAPI 加密。
- 防止 ZIP/RAR 路径穿越，禁止解压内容写出目标目录。
- 删除可游玩目录前必须确认最新存档已经完成有效外部备份。
- 复制和备份完成后执行文件清单及 Hash 校验。
- 磁盘空间不足时提前停止。
- 文件被占用时保留数据并显示具体错误。
- 支持 Windows 长路径、中文和特殊字符路径。
- 机械盘默认低并发顺序读取。
- 软件异常退出后能够识别并恢复未完成任务状态。
- 计划任务与主程序同时运行时必须避免数据库并发写入冲突。

- 能够完成普通归档、特殊归档和版本切换。
- 能够执行手动备份和完整 `GameSave` 每日计划备份。
- 能够在备份盘离线后补执行备份。
- 能够查看外部 ZIP 内容和人工恢复说明。
- 所有删除行为符合确认次数、回收站和永久删除规则。
- 归档状态不依赖游戏目录删除结果。
- 关键任务失败时不丢失原始文件、存档或有效备份。

## 26. 项目目录与发布要求

### 26.1 项目目录

项目根目录固定为：

```text
D:\workspace\GameManagement
```

后续生成的项目源码、工程配置、构建脚本、测试代码和项目文档均存放在该目录下。不得将项目源码生成到其他临时工作目录后作为正式项目使用。

建议的源码目录结构为：

```text
D:\workspace\GameManagement\
├─ docs/
├─ src/
├─ tests/
├─ tools/
├─ build/
└─ 游戏管理软件需求与实施计划_v1.0.0.md
```

实际目录结构应在技术栈确定后进一步细化。

### 26.2 可执行文件

最终发布产物必须是一个可以独立运行的单文件 EXE：

```text
GameManagement.exe
```

发布时不得要求用户额外安装运行时、解压工具或其他程序依赖。ZIP、RAR 解压所需组件及程序运行依赖应一并打包进单文件 EXE，或由 EXE 在内部以不需要用户干预的方式使用。

### 26.3 默认运行目录

EXE 启动时，以 EXE 文件所在目录作为默认软件数据根目录，在当前目录下创建软件运行所需的数据文件和日志目录，不使用源码目录中的固定绝对路径。

建议默认结构：

```text
GameManagement.exe
data/
├─ game-management.db
├─ settings/
└─ cache/
logs/
├─ application/
├─ task/
└─ error/
```

具体要求：

- 首次运行时自动创建缺失的数据目录和日志目录。
- 配置、数据库、任务状态和运行日志默认保存在 EXE 所在目录。
- 路径解析必须基于 EXE 所在目录，不得依赖进程启动时的工作目录。
- 用户从快捷方式、资源管理器、命令行或计划任务启动时，均应定位到相同的数据目录。
- 软件应检查当前目录的写入权限；不可写时明确提示，不得静默改写到其他目录。
- 软件升级时应保留已有数据库、配置和日志。
- 用户配置的游戏盘、扫描目录和外部备份目录不受默认运行目录限制。
- Windows 计划备份组件必须读取 EXE 所在目录下的同一套配置和数据库。

### 26.4 构建验收要求

- 构建产物为单个 EXE 文件。
- 在未安装开发环境和第三方解压软件的干净 Windows 环境中能够启动。
- 将 EXE 复制到新的可写目录后能够首次运行并自动创建数据和日志目录。
- 从其他工作目录启动 EXE 时，数据仍创建在 EXE 所在目录。
- 目录不可写时显示中文错误提示并安全退出。
- EXE 文件移动后能够使用新目录中的数据；不得继续引用旧发布目录的绝对路径。

## 27. 版本说明

版本 1.0.0 为当前已确认需求的首个正式需求基线。后续需求变更应记录版本号、变更原因、影响范围和确认结果，避免直接覆盖既有规则。
