| 项目 | 内容 |

## 1. 项目目标

本项目用于统一管理硬盘中的单机游戏原始压缩文件、可游玩目录和游戏存档。

机械硬盘只保存下载得到的原始 ZIP、RAR、7z 压缩文件及其分卷；用户准备游玩时，软件将所选原始文件复制到指定的固态游戏盘，完成两级解压、游戏目录识别、启动程序识别和存档恢复。游戏归档时，软件保存游戏目录内变化文件以及按游戏配置启用的 Windows 系统目录存档，并在本地存档清单与 Hash 校验完成后清理可游玩目录，释放固态空间。

软件提供 Windows 图形化界面，所有复制、扫描、解压、Hash、备份和清理任务均需要显示状态、进度和错误信息。

## 2. 项目范围

### 2.1 支持范围

- 仅支持 Windows 系统。
- 支持配置多个游戏盘，每次准备、归档或特殊归档前选择目标游戏盘。
- 支持添加多个原始游戏文件扫描路径。
- 支持 ZIP、RAR、7z 识别与解压，并支持 `.7z.001/.7z.002` 形式的 7z 分卷。
- ZIP、RAR 和 7z 解压能力随软件提供，不依赖用户另行安装解压软件。
- RAR 只要求解压，不要求创建 RAR 文件。
- 备份文件统一创建为无密码 ZIP。
- 支持多层目录、两级压缩、错误扩展名和分卷压缩文件。
- 支持一个游戏配置多个版本，但同一时间只允许一个版本处于可游玩状态。
- 支持游戏目录内存档和经用户确认的 Windows 常见系统目录存档。
- 支持正常退出快照与异常退出快照。
- 支持每日计划备份和手动备份。

### 2.2 暂不支持

- 非 Windows 系统。
- Steam、Epic 等游戏平台安装游戏管理。
- 模拟器游戏专项管理。
- 一个游戏配置多个主启动程序。
- 同一个游戏的多个版本同时保持可游玩状态。
- 创建 RAR 备份。
- 多个外部备份目标目录。
- 在软件内自动恢复外部 ZIP 备份。
- 对游戏进程注入 DLL 或修改进程代码。

## 3. 术语与目录定义

### 3.1 术语

| 术语 | 定义 |
|---|---|
| 原始文件 | 机械盘中下载得到的 ZIP、RAR、7z、分卷文件或包含这些文件的目录 |
| 游戏盘 | 用户配置的固态工作盘，用于保存临时文件、可游玩目录和本地存档 |
| 可游玩目录 | 已完成解压并包含有效主游戏 EXE 的目录 |
| 本地存档 | 保存在游戏盘 `GameSave` 目录中的活动存档、快照和清单 |
| 外部备份 | 保存在单独备份目录中的无密码 ZIP 文件 |
| 文件基线 | 首次解压完成后记录的干净游戏目录文件信息 |
| 普通归档 | 保存存档后处理软件正常准备的可游玩目录 |
| 特殊归档 | 对用户已有的混乱游戏目录与干净基准目录进行比较和存档提取 |

### 3.2 游戏盘目录

每个游戏盘使用独立工作根目录：

```text
{游戏盘工作根目录}/
├─ Games/
│  └─ {游戏ID}/
│     ├─ .prepare/
│     └─ content/
├─ GameTemp/
│  └─ （仅兼容旧版本遗留任务）
├─ GameSave/
│  └─ {游戏ID}/
└─ GameSaveTemp/
   └─ {游戏ID}/{版本ID}/
```

游戏 ID 和版本 ID 使用不含中文的随机 UUID。

## 4. 配置管理

### 4.1 扫描路径

- 支持添加、删除、启用和禁用多个扫描路径。
- 所有扫描路径需要持久化保存。
- 支持手动重新扫描。
- 显示路径状态、所属磁盘、剩余空间和最近扫描时间。
- 路径失效时显示错误并提供重新定位功能。

当前实现支持在扫描路径列表中查看有效状态、所属磁盘、剩余空间和最近扫描时间，并通过独立按钮启用、禁用或重新定位路径。

### 4.2 游戏盘

每个游戏盘配置包含：

- 显示名称。
- 工作根目录。
- 是否启用。
- 是否默认选中。
- 最低剩余空间阈值。
- 磁盘容量、剩余空间和磁盘类型。

执行准备、再次准备、存档复制或特殊归档前必须选择游戏盘。界面显示预计所需空间和操作后预计剩余空间。

当前实现支持配置多个游戏盘、默认游戏盘、启用状态和最低保留空间。设置页面显示路径状态、磁盘类型、总容量、剩余空间和最低保留空间；准备游玩时默认游戏盘排在选择列表首位，但仍由用户最终确认。设置页同时提供 EXE 忽略清单管理，支持新增、编辑、删除并持久化具体 EXE 文件名；扫描时按完整文件名忽略大小写匹配，命中的文件不用于确定游戏目录，也不进入启动文件候选。设置页提供“自动选择 / WinRAR / 内置解压”引擎选项并持久化：自动模式优先使用从 Windows 注册表或标准安装目录检测到的 WinRAR，未检测到时回退内置解压；强制 WinRAR 模式不可用时终止并提示，禁止静默回退。

### 4.3 外部备份目录

- 当前版本只支持一个外部备份目录。
- 记录卷标、卷序列号、最近盘符和相对目录，降低盘符变化影响。
- 备份盘离线时将任务标记为“等待目标磁盘”。
- 目标磁盘重新上线后自动补执行等待中的计划备份。

## 5. 文件扫描与游戏添加

### 5.1 文件扫描列表

扫描列表显示：

- 文件或目录名称。
- 完整路径。
- 文件类型。
- 文件大小。
- 修改时间。
- 所属磁盘。
- 是否疑似压缩文件。
- 来源形式：直接 ZIP/RAR/7z 压缩文件或分卷入口，或内部包含压缩文件的目录。
- 目录来源的压缩包状态；文件扫描阶段显示“准备时扫描”，不在此阶段递归读取目录内容。
- 是否已经加入游戏库。
- 路径是否有效。
- “是否已经加入游戏库”和“路径是否有效”属于只读展示字段，界面必须使用单向绑定，不得尝试回写模型属性。

支持按名称、路径、类型、磁盘和添加状态过滤，并支持隐藏已加入游戏库的文件夹。

扫描得到的待添加文件列表默认按文件或目录的最后修改时间倒序排列；修改时间相同时按名称升序排列，以保证显示顺序稳定。

为避免机械硬盘扫描时因递归读取大量文件及压缩包头尾而长时间卡顿，游戏库文件扫描采用以下边界：

- 只枚举每个已启用扫描路径的直属文件夹。
- 只枚举每个已启用扫描路径直属且扩展名为 `.zip`、`.rar`、`.7z` 或匹配 `.7z.NNN` 分卷格式的文件。
- 不在游戏库扫描阶段递归进入候选文件夹，也不读取其他直属文件的文件头或尾部。
- 候选文件夹内部的 ZIP、RAR、7z、7z 分卷、错误扩展名压缩文件及混淆文件，延迟到“准备游玩”阶段递归识别。
- 扫描过程在后台执行，界面显示已完成数量、总数量、百分比和当前处理路径。
- 扫描完成前禁用主界面其他操作，并禁止关闭主窗口；扫描成功或失败结束后恢复操作。

### 5.2 添加游戏

用户判断某个文件或目录属于游戏后，可将其添加至游戏库：

- 生成随机 UUID 游戏 ID。
- 支持在扫描候选列表中多选并批量添加；来源指纹计算及持久化在后台运行，主窗口保持可操作，进度、取消和结果写入任务中心。
- 添加时不再要求用户输入游戏名称。
- 文件夹候选直接使用文件夹名称作为游戏名称；ZIP、RAR、7z 候选使用去除扩展名后的文件名称作为游戏名称，`.7z.NNN` 分卷入口同时去除 `.7z.NNN` 后缀。
- 两种原始来源均可添加：直接选择单个 ZIP/RAR/7z 文件或 7z 分卷入口，或选择内部包含一个或多个支持压缩文件的目录。
- 游戏和版本数据需要持久化保存来源形式，后续准备流程不得假定来源一定是目录或一定是文件。
- 不修改实际文件或目录名称。
- 保存原始路径、大小、修改时间和文件指纹。
- 批量添加时自动创建首个游戏版本，版本名称统一为“初始版本”。

### 5.3 游戏版本

一个游戏允许包含多个版本，每个版本独立保存：

- 版本 ID。
- 版本名称和备注；首个批量添加版本默认为“初始版本”，后续新增版本名称由用户填写。
- 原始压缩文件路径及指纹。
- 用户选择的压缩文件或分卷组。
- 第一次与第二次解压规则和密码。
- 游戏根目录识别结果。
- 主游戏 EXE 相对路径。
- 干净文件基线。
- 最后游玩时间。

同一游戏只允许一个版本处于可游玩状态。切换版本前必须完成当前版本归档。

阶段二实现约束：

- 添加游戏或新增版本时，在阻塞式进度窗口中计算来源文件数量、总体积、最后修改时间和 SHA-256 指纹；目录来源使用内部全部文件的相对路径、大小和内容 SHA-256 生成稳定聚合指纹。
- 旧数据缺少来源指纹时，在首次准备游玩前自动补充并持久化。
- 游戏详情支持在同一个编辑窗口中同时修改显示名称和备注，备注允许留空，不修改硬盘上的实际名称。
- 版本管理支持添加用户命名版本、编辑版本名称与备注、查看来源状态、将版本设为当前版本、打开原始位置和重新定位来源。
- 当前存在可游玩目录时禁止切换版本；必须先完成后续阶段定义的归档流程。
- 重新定位时重新校验 ZIP/RAR 文件或压缩文件目录，计算新来源元数据，并展示文件名、总体积、修改时间和 SHA-256 的匹配结果，由用户最终确认是否替换。
- 游戏列表、游戏详情和版本列表显示原始路径是否有效。

阶段二当前实施进度：多个扫描路径及持久化、直属候选扫描、名称/路径/类型/磁盘/添加状态筛选、候选只读状态单向绑定、后台批量添加、游戏名称与备注维护、多版本新增与切换、来源元数据持久化、路径失效显示和人工重新定位已实现；批量添加计算来源指纹时不再禁用主窗口，进度和取消仍由独立进度窗口及任务中心提供。分卷组校验已经在阶段四补充完成。扫描结果首次渲染引发的只读属性双向绑定异常已完成回归修复，并补充全局未处理异常日志。

## 6. 游戏列表与详情

游戏列表至少显示：

- 游戏图标。
- 游戏名称或备注。
- 当前版本。
- 原始文件路径。
- 当前游戏盘。
- 是否存在本地存档。
- 是否存在有效外部备份。
- 上次游玩时间。
- 游戏状态。
- 目录清理状态。

游戏库首先将归档状态或游戏状态为“已归档”的记录统一排在未归档游戏之后；未归档组和已归档组内部均按加入游戏库时间倒序、上次游玩时间倒序、游戏名称升序排列。状态变化后立即重新排序，不要求用户手动刷新。旧游戏记录缺少加入时间时，使用其最早版本添加时间兼容迁移，无版本记录时排在同组具有有效加入时间的记录之后。

游戏详情页提供：

- 在游戏库选择游戏后点击“查看详情”进入；双击游戏列表中的游戏直接启动该游戏。游戏库列表支持右键菜单，右键所在行自动成为当前游戏；菜单完整提供游戏详情中的准备、编辑、启动、归档、备份、存档、版本、密码、识别、重新定位、打开和删除功能，并复用游戏详情的同一操作逻辑与安全确认。
- 游戏库右键菜单执行具体功能时不显示游戏详情窗口，直接打开对应功能窗口或执行操作；“查看详情”按钮进入完整详情页，双击直接执行启动。右键操作复用详情逻辑时只允许创建不进入可视树的内存操作宿主，禁止调用 `Show`、生成透明/黑色详情窗口或在任务栏创建窗口；进度窗口和结果提示统一由主窗口持有。通过右键执行准备、编辑、版本、模板、系统存档、归档、存档、重新定位或删除等会改变游戏信息的操作时，每次持久化后同步刷新游戏库行；删除主记录后立即移除该行，异步准备完成后无需用户手动刷新即可显示最新状态、路径、版本和存档信息。
- 游戏详情和游戏库右键菜单提供“选择解压模板”，选择结果以模板 ID 持久化到游戏记录中；游戏库右键入口使用二级菜单直接列出“不使用模板”和全部可用模板，当前模板显示勾选状态，点击子项立即保存，不再弹出选择窗口。设置页提供统一的解压流程模板管理入口。
- 游戏库列表支持扩展多选。选中多个游戏时，右键菜单只开放“选择解压模板”“准备游玩”和“归档游戏”：模板一次应用到全部所选游戏；准备与归档按列表选择顺序进入串行后台队列，避免并发复制、解压或归档争抢磁盘，主窗口保持可操作。批量后台任务使用无界面的进度接收器，不实例化 XAML 内容、不设置 Owner，也不执行窗口显示、隐藏、关闭或完成后的前台激活流程；只更新主窗口顶部批次总进度和任务中心，确实需要人工选择时才显示对应选择或确认窗口。批量操作中的同类提示只在首次出现时询问，用户选择应用到本批次全部游戏，包括游戏盘、覆盖与兼容风险、临时目录清理、历史流程、第一/第二压缩文件、解压密码、启动文件、无清单归档、系统存档存在性和归档后目录清理；复用的磁盘、文件名、相对路径或密码在当前游戏不存在、不可用或验证失败时，才允许针对该游戏重新询问。不同游戏独有的存档文件树和系统存档目录路径仍需分别确认。

- 准备游玩。
- 启动游戏。
- 打开游戏目录。
- 归档游戏。
- 特殊归档。
- 手动备份。
- 存档目录设置。
- 存档与快照查看。
- 版本管理。
- 原始文件重新定位。
- 独立的原始文件删除按钮。

## 7. 状态模型

建议使用以下游戏状态：

```text
待配置
未准备
复制中
第一次解压中
第二次解压中
等待确认游戏目录
可游玩
运行中
等待确认系统存档
归档中
已归档
版本切换失败
源文件失效
操作失败
```

归档状态与目录清理状态相互独立。本地存档保存、清单及 Hash 校验完成后即可标记“已归档”，不要求外部 ZIP 备份；游戏目录是否成功进入回收站不影响归档状态。

目录状态示例：

```text
目录存在
等待清理
已移入回收站
清理失败
```

## 8. 首次准备流程

### 8.1 准备检查

1. 用户选择游戏版本和目标游戏盘；可用游戏盘只有一个时自动选择，不显示人工选择窗口。
2. 检查原始文件路径及格式。
3. 检查同一游戏是否已有其他可游玩版本。
4. 估算复制和解压空间。
5. 检查游戏盘剩余空间。
6. 创建 `Games/{游戏ID}/.prepare` 准备目录；失败或取消时保留该目录，但不得写入可游玩路径。

### 8.2 第一次解压

1. 将机械盘原始文件复制到游戏盘临时目录。来源是单个 ZIP/RAR/7z 或分卷入口时复制完整分卷组；来源是目录时复制该目录及其内部文件。
2. 来源是单个压缩文件或分卷入口时直接将该文件或分卷组作为第一次解压候选；来源是目录时递归扫描其中所有 ZIP、RAR、7z 和分卷组。
3. 只有一个压缩文件或分卷组时自动选择；存在多个候选时由用户手动选择。
4. 用户输入第一次解压密码，或读取已保存密码。
5. 解压到第一阶段目录并记录结果。

分卷缺失时允许用户确认后尝试解压一次；失败后停止，不自动重试，也不继续后续步骤。

阶段四分卷实现规则：

- 将 `.partNN.rar`、`.rar/.r00/.r01`、`.zip.001/.zip.002`、`.z01/.z02/.zip` 和 `.7z.001/.7z.002` 识别为分卷组，在选择窗口中以一个候选项展示入口文件、分卷数量、总体积和缺失状态。
- 7z 数字分卷按卷号从 `.001` 开始连续检查，并读取首卷 7z 起始头中的下一头偏移与长度，结合首卷大小推算完整归档最低长度和应有卷数，从而识别只有 `.001`、但实际缺少末尾 `.002` 等分卷的情况；解压前在游戏盘临时目录中按顺序合并为临时 `.7z` 文件，解压或验证结束后自动删除临时合并文件，不修改机械盘原始分卷。即使解析归档头失败，也必须由软件持有并释放临时文件流，避免异常后临时合并文件持续被进程占用。
- 用户从单个分卷入口文件添加游戏时，复制、空间估算和来源聚合指纹覆盖同组全部已存在分卷，不得只处理入口文件。
- 根据当前可见的最大卷号检查从首卷到最大卷号之间的连续性，并检查旧式 RAR 的 `.rar` 入口或旧式 ZIP 的最终 `.zip` 文件。
- ZIP、RAR 等无法从首卷取得完整归档最低长度时，仍只能检查现有最大卷号之前的连续性；7z 数字分卷可额外利用首卷头部推断末尾缺失卷。界面统一将检查结果表述为“可推断的缺失卷”。
- 检测到缺失卷时列出文件名并要求人工确认。确认后只执行一次解压；无论密码、格式还是分卷错误导致失败，都立即停止，不重新询问密码，也不继续搜索 EXE。

### 8.3 第二层压缩识别

- 递归扫描第一次解压结果。
- 检查文件头部和尾部区域中的 ZIP、RAR 标识，不仅依赖扩展名。需要兼容在 MP4 等伪装数据之后内嵌 ZIP，并在文件末尾保留 ZIP 结束目录的混合文件。
- 对带有 MP4 等前置伪装数据的混合文件，先定位真实 ZIP/RAR 起始签名，再根据最终解压引擎分流：内置解压需要将起始签名之前的伪装前缀排除并生成规范化临时压缩文件；WinRAR 或自动模式实际检测到 WinRAR 时，直接将原始混淆文件交给 WinRAR，不剪切、不生成规范化大文件副本，以减少大型游戏的磁盘读写和等待时间。仅修改扩展名不足以保证内置解压组件能够识别。
- 大文件优先作为候选，但不得仅凭文件大小自动确定。
- 识别出真实格式后，在第一次解压产生的临时目录中将文件改为 `.zip` 或 `.rar`，不修改机械盘原始文件。
- 存在多个候选时由用户手动选择。
- 只有一个第二次解压候选时自动选择，不显示人工选择窗口。
- 如果通过文件头部和尾部仍然无法识别压缩文件，选择第一次解压结果中最大的未识别文件，先改名为 `.zip` 尝试带密码解压；失败后再改名为 `.rar` 尝试带密码解压。
- ZIP、RAR 两种尝试均失败时停止准备流程并保留临时目录，不得直接进入 EXE 搜索。

### 8.4 第二次解压

- 第二次密码独立于第一次密码，并与具体压缩文件指纹绑定。
- 解压失败时允许重新输入密码或重新选择文件。
- 成功后进入游戏目录与主 EXE 识别。
- 选择 WinRAR 或自动检测到 WinRAR 时，通过无窗口子进程执行静默解压或密码验证，自动传入当前步骤的模板密码、历史密码或用户输入密码；监听退出码并转换为中文错误，任务取消时终止该 WinRAR 子进程。第二层混淆压缩文件直接使用原始文件，不执行内置引擎专用的前缀剪切和规范化复制。WinRAR 为用户自行安装的外部商业软件，项目不得捆绑其二进制文件。
- WinRAR 命令行接口要求密码作为进程参数传递；持久化数据仍使用 DPAPI 加密，但解压进程运行期间，本机具有足够权限的其他程序可能读取该瞬时命令行参数，设置界面和文档必须保留此安全说明。

### 8.5 游戏目录和主 EXE 识别

从最终解压结果开始按目录层级和目录名称稳定排序进行递归搜索。找到第一个不属于排除名单的有效 EXE 后，将该 EXE 所属文件夹标记为最终游戏目录，并停止继续搜索其他目录。用于确定游戏目录时排除常见辅助程序，例如：

```text
UnityCrashHandler.exe
UnityCrashHandler64.exe
CrashReportClient.exe
UE4PrereqSetup_x64.exe
UEPrereqSetup_x64.exe
unins000.exe
uninstall.exe
setup.exe
vc_redist.x64.exe
DXSETUP.exe
```

确定游戏目录后，列出该目录直属且未命中 EXE 忽略清单的 `.exe` 文件，以及文件名为 `index.html` 的网页入口，不递归展示子目录文件。只有一个启动候选时自动选择；存在多个候选时准备阶段不弹窗并允许完成准备，首次启动时再由用户人工确认。EXE 忽略清单由设置页维护并持久化，按完整文件名忽略大小写匹配；命中的文件既不用于确定游戏目录，也不进入候选列表。每个游戏版本保存一个启动文件相对路径；再次准备同一版本时优先在新解压结果中读取并验证该相对路径，文件存在且类型有效时自动复用。其他插件或程序由用户通过“打开游戏目录”自行启动。

识别成功后：

- 提取主 EXE 图标，并将图标缓存相对路径同时保存到游戏及对应版本记录；游戏库列表与详情统一从持久化图标信息读取，归档清理不得删除图标信息。旧数据存在按游戏标识命名的图标缓存时自动补回持久化路径。
- 没有游戏备注时，可以使用目录名称作为候选备注，由用户确认。
- 在 `Games/{游戏ID}/.prepare` 中完成复制、两次解压和基线计算，再以同盘目录移动方式提交到 `Games/{游戏ID}/content`，不得为最终目录再次全量复制。
- 保存游戏根目录和主 EXE 相对路径。
- 将用户确认的启动文件相对路径同时保存到具体游戏版本，旧数据在启动文件仍可用时自动补充到当前版本。
- 建立干净文件基线。
- 自动永久删除成功任务的 `.prepare`，无需确认；失败目录可由任务中心在严格路径校验后清理。

当前实施进度：已经实现游戏盘选择、压缩条目内容感知空间估算、元数据不可读时按来源体积四倍安全回退、最低剩余空间检查、原始来源及完整分卷组复制、两次压缩文件或分卷组人工选择、可推断分卷缺失检查与一次尝试确认、ZIP/RAR 头尾区域识别、按解压引擎拆分混合文件处理（内置引擎生成规范化临时文件，WinRAR 直接读取原始混淆文件）、错误扩展名临时改名、最大未识别文件依次按 ZIP/RAR 尝试、DPAPI 密码保存与管理、历史解压流程重放、ZIP/RAR 解压、多启动文件延迟确认、同盘目录快速提交、EXE 图标提取和 SHA-256 文件基线。空间估算会分别计算来源复制量、首层展开量、二次解压与最终目录预留以及安全余量；加密或混淆的二次压缩内容在完成首层解压前无法得到绝对精确展开体积，最终仍以目标游戏盘实际剩余空间为准。

阶段五补充实现：再次准备同一版本时优先复用该版本已保存且验证有效的启动文件相对路径，仅在无法定位时回退人工选择。启动候选在不改变“按稳定顺序找到第一个有效 EXE 所属目录”规则的前提下进行评分排序。EXE、与目录同名、目录内体积最大的 EXE、包含 `game`/`launch`/`start` 关键词以及版本信息与目录名相关时提高评分；辅助程序排除名单降低评分。评分只用于推荐排序，最终仍由用户人工确认。自动识别失败或推荐不正确时允许在本次解压结果范围内手动选择 EXE 或 `index.html`。选择完成后提取 EXE 图标并保存到程序 `data/cache/icons`，图标相对路径同时写入游戏与当前版本；游戏库列表和详情从持久化路径显示图标，归档后仍保留，旧数据可根据现有缓存自动迁移。网页入口不生成 EXE 图标。

## 9. 密码管理

- 第一次和第二次密码分别保存。
- 密码与具体游戏版本及压缩文件指纹绑定。
- 使用 Windows DPAPI 加密存储。
- 解压密码输入窗口以明文显示当前输入内容，不使用掩码隐藏；密码管理界面中的已保存凭据仍默认隐藏，用户可以主动查看。
- 解压密码输入框左侧提供密码历史弹窗，以竖向列表显示当前 Windows 用户曾经输入并保存的密码；存在历史密码时弹窗默认展开，用户可通过“密码历史”按钮收起或重新展开；双击任意历史密码后，立即填入并提交解压。
- 密码历史跨游戏和版本持久化保存，使用当前 Windows 用户的 DPAPI 加密，按解密后的完整密码精确去重；重复使用已有密码时只更新最近使用时间，不新增重复记录，空密码不加入历史。
- 支持修改、删除和重新验证。
- 支持按来源渠道建立解压流程模板，模板分别保存第一次、第二次解压密码并使用当前 Windows 用户 DPAPI 加密。游戏保存所选模板 ID；准备时若没有对应压缩文件的专用历史密码，则按步骤读取模板密码，验证失败后回退人工输入并保存文件级密码。仍被游戏引用的模板禁止删除。
- 日志不得记录明文密码。
- 外部存档 ZIP 不包含游戏解压密码。

密码管理界面按游戏版本和第一次、第二次解压步骤显示记录，默认不展示明文。用户主动点击后使用当前 Windows 用户的 DPAPI 解密查看；修改后标记为等待验证；删除需要一次人工确认。能够定位压缩文件时支持读取首个文件内容重新验证，无法定位第二次临时压缩文件时提示将在下次准备游玩成功解压后自动更新验证时间。

每次成功解压保存压缩文件显示名称、相对路径、步骤序号、文件指纹和最近验证时间。日志禁止记录明文密码。

### 9.1 历史解压流程重放

- 每个版本保存第一次和第二次解压使用的相对路径、识别格式，以及第二次是否使用“最大未识别文件改名”降级流程。
- 再次准备同一版本时，如果记录的相对路径仍存在，第一次解压前提示是否重放历史选择和已保存密码，并提供“后续解压步骤均自动使用历史选择和已保存密码”复选框。用户确认重放且勾选后，本次准备任务中后续存在有效历史记录的解压步骤不再逐次询问；未勾选时，第二次解压仍单独提示是否重放。
- 用户拒绝重放时返回当前压缩文件或分卷组人工选择，不强制使用历史记录。
- 第二次曾使用最大文件降级流程时，按上次成功的 ZIP 或 RAR 格式直接重放；历史文件不存在或用户拒绝时重新执行候选扫描和人工选择。

## 10. 文件基线与 Hash

文件基线至少包含：

- 游戏 ID、版本 ID。
- 相对路径。
- 文件大小。
- 修改时间。
- SHA-256。
- 文件类别。

比较时先使用路径、大小和修改时间快速判断，必要时计算 SHA-256。首次准备完成后必须成功保存基线，才能自动清理临时目录。

归档候选默认排除满足以下条件的文件：

- 大于或等于 100 MB。
- 首次基线中已经存在。
- 疑似游戏资源包。
- 未被用户明确确认为存档。

已确认的存档文件不受 100 MB 排除规则限制。小型日志、缓存、截图和配置文件不自动排除，统一交由用户确认。用户排除规则只针对具体文件，不支持目录、通配符或正则表达式。

阶段六实现约束：比较时先按相对路径、大小和修改时间判断，仅对疑似新增或修改内容计算 SHA-256；删除文件使用首次基线记录生成候选。首次基线中已存在、大小大于或等于 100 MB、疑似资源包的文件标记为默认排除，但仍显示在候选列表并允许用户明确确认。已经确认过的具体存档文件不再应用上述默认排除规则；确认并存入 `GameSave/current` 后，后续退出扫描改为使用来源文件与 current 副本的大小、修改时间进行快速比较，未变化时不再计算 Hash、生成候选或重复归档，只有元数据变化时才进入归档及最终 Hash 完整性校验。用户排除的具体文件在后续扫描中自动标记为已排除。基线记录补充游戏 ID 和文件类别，旧数据继续按版本 ID 兼容读取。

## 11. 游戏运行监控

### 11.1 进程判断

- 游戏运行与结束完全以主游戏 EXE 状态为准。
- 主 EXE 启动后状态变为“运行中”。
- 主 EXE 退出后本次运行结束。
- 插件、修改器和其他进程不影响运行状态。
- 不进行 DLL 注入，采用进程生命周期监听和文件系统变更通知。

阶段五实现约束：启动时保存主进程 ID、开始时间和上次游玩时间；进程退出后保存退出时间、退出代码和本次持续时长，并将状态恢复为“可游玩”。软件关闭后游戏进程仍在运行时保留进程信息，下次启动通过进程 ID、启动时间和可访问时的 EXE 路径恢复监听；记录已经失效时自动修正过期的“运行中”状态。重复启动同一游戏的已监听主进程被禁止。系统存档目录扫描和文件系统变化监控按阶段七实施。

### 11.2 首次系统存档扫描

游戏详情和游戏库行右键菜单均提供“此游戏存在 Windows 系统目录存档”复选框，两个入口同步显示并修改同一状态。该选项默认不勾选，新添加游戏及旧数据中缺少此字段的记录均默认只使用游戏目录内存档：启动前和退出后跳过 Windows 常见目录及已配置系统目录扫描，归档时也不要求完成首次系统存档扫描；用户明确勾选后才启用系统目录扫描流程。

游戏第一次运行前，对以下常见目录建立快照：

```text
%APPDATA%
%LOCALAPPDATA%
%USERPROFILE%\AppData\LocalLow
%USERPROFILE%\Documents
%USERPROFILE%\Saved Games
```

首次公共目录扫描应跳过明显属于其他软件或系统组件的目录，内置排除 `Microsoft`、`Tencent`、`AMD`、`NVIDIA`、`NVIDIA Corporation`、`Adobe`、`Google`、`Mozilla`、`Discord`、`Temp`、`CrashDumps` 和 `ConnectedDevicesPlatform`；排除规则不应用于用户后续明确配置的存档目录。

主 EXE 退出后只比较目录定位信息，不读取 Hash 或生成逐文件候选。软件优先定位运行期间新建且与游戏名称、备注、主 EXE 文件名或可游玩目录名匹配的目录，路径各级目录名及文件名均参与匹配，匹配时忽略空格、下划线、横线、点和括号，并过滤隐藏文件及隐藏目录。例如 `NTR Soccer` 或 `NTR_Soccer.exe` 可匹配并保存 `LocalLow\Hizure\NTR_Soccer`。首次匹配成功后仅持久化存档文件夹路径，等待归档时再扫描该文件夹及其内容；首次匹配失败后标记首次尝试完成，后续启动不再自动执行公共目录匹配或系统存档文件扫描。

### 11.3 后续监控

- 首次定位尝试完成后不再扫描全部常见目录，也不在每次游戏启动和退出时扫描已定位目录。
- 已定位的一个或多个存档目录只在归档时扫描。
- 首次自动定位失败时，归档前询问用户是否存在系统存档；用户确认存在后人工选择存档文件夹，用户确认不存在时本次归档只处理游戏目录存档。
- 用户可手动添加、修改、禁用和删除目录。
- 同一个系统目录允许关联多个游戏。
- 共享目录中的文件归属不明确时必须再次由用户确认。

### 11.4 存档候选预览

- 候选选择界面使用带复选框的文件树而非平面列表，支持同时勾选多个文件或目录。
- 候选选择界面既可能由用户以模态方式打开，也可能在游戏退出后以非模态方式自动弹出；关闭按钮在两种打开方式下都必须立即关闭窗口。
- 勾选目录等同于勾选该目录及其全部下级文件；系统存档目录在归档时展开并生成可选文件树。
- 隐藏文件与隐藏目录不进入自动匹配和归档候选树。

支持：

- 查看文件路径、大小、修改时间和变化类型。
- 预览文本和常见图片。
- 使用系统默认程序打开文件。
- 在资源管理器中定位文件。
- 确认为存档。
- 排除具体文件。

## 12. 本地存档与快照

建议目录：

```text
GameSave/{游戏ID}/
├─ current/
├─ normal-snapshots/
├─ abnormal-snapshots/
└─ manifest.json
```

- 只有存档发生变化时才创建快照。
- 正常退出快照默认保留三个。
- 异常退出快照单独标记并默认保留三个。
- 异常快照不作为默认恢复首选。
- 主 EXE 异常退出且存档变化时立即创建异常快照。
- 用户允许手动删除快照，删除前确认一次。
- 超过保留数量时不自动删除，标记为建议清理并提醒用户确认。

阶段六实现约束：主程序退出代码为 `0` 时使用正常候选，非 `0` 或无法取得退出代码时使用异常候选。已确认规则中的新增或修改文件在主程序退出后自动更新 `current` 并立即创建对应快照；首次出现的文件、默认排除项和删除变化弹出候选页面，由用户确认或按具体文件排除。确认删除变化前必须进行一次人工确认。快照处理先在 `GameSave/{游戏ID}` 下构建临时 current 和 `.partial` 快照，完成文件清单与 SHA-256 校验后再替换 current；失败时回滚原 current，临时目录自动清理。内容指纹没有变化时不生成重复快照。

管理页面支持查看当前存档、正常和异常快照、清单、校验状态、退出代码与建议清理标记；支持文本和常见图片预览、系统程序打开及资源管理器定位。恢复由用户人工执行，页面提示按相对路径复制及跨版本兼容风险。每类最新三个快照标记为保留，超过三个的旧快照只标记为建议清理，禁止自动删除；人工删除快照需要一次确认。

同一个游戏的不同版本默认共享存档。向不同版本恢复前提示兼容风险，并保留来源版本信息。

## 13. 跨游戏盘存档复制

再次准备时选择了不同游戏盘，应将主存档复制到目标盘：

1. 复制到目标临时目录。
2. 校验临时副本。
3. 目标位置已有存档时，每次询问用户是否覆盖。
4. 用户确认后替换目标存档。
5. 再次校验目标存档。
6. 来源游戏盘存档保留。
7. 将目标副本设置为当前主存档。

该流程属于游戏盘间存档复制，不属于外部归档备份。

## 14. 再次准备与版本切换

机械盘不保存完整可游玩目录，所以再次游玩必须重新解压：

1. 选择目标版本和游戏盘。
2. 检查原始文件，失效时重新定位。
3. 使用保存的压缩文件组、密码和解压规则重新执行两级解压。
4. 使用历史规则识别游戏根目录和主 EXE。
5. 恢复共享存档。
6. 自动清理成功任务临时目录。

切换版本前必须：

1. 结束当前主游戏 EXE。
2. 保存并校验最新存档。
3. 将当前版本标记为已归档；外部 ZIP 备份不作为归档或版本切换的前置条件。
4. 询问后将当前游戏目录移入回收站。
5. 准备目标版本。

新版本准备失败时只进行状态标记和错误记录，不自动从回收站恢复旧版本。

## 15. 普通归档

归档成功条件：

- 游戏目录内存档收集成功。
- 游戏标记为存在系统存档时，已配置的系统目录存档收集成功；标记为不存在时跳过此项。
- 本地存档快照创建并校验成功。
- 存档清单写入成功。

游戏已经准备完成且当前未运行时，如果没有可校验的存档快照和清单，软件必须弹出一次风险说明；无论此前是否运行过，用户明确确认后均允许直接归档，用户取消时保持可游玩状态。主游戏程序仍在运行时始终禁止归档。

普通归档不检查外部 ZIP 是否存在、是否匹配最新内容或是否校验成功，也不会自动创建外部 ZIP。手动备份和每日计划备份保留为可选的异盘灾备能力。

满足归档条件后立即将游戏状态标记为“已归档”。随后询问一次是否将游戏目录移入回收站。目录删除成功与否不影响归档状态。

已归档但目录仍存在时允许继续启动游戏；启动后恢复为“可游玩/运行中”，并重新监控存档变化。

## 16. 特殊归档

特殊归档流程：

1. 选择游戏和版本。
2. 选择用户已有的混乱游戏目录。
3. 选择操作游戏盘。
4. 在 `GameSaveTemp` 中从原始压缩文件构建干净基准目录。
5. 与混乱游戏目录进行相对路径、大小、时间和 Hash 比较。
6. 展示新增、修改、缺失及默认排除文件。
7. 用户确认存档候选。
8. 创建并校验本地存档快照。
9. 标记游戏已归档，不检查或创建外部 ZIP 备份。
10. 询问一次后将混乱游戏目录移入回收站。
11. 成功任务自动永久清理 `GameSaveTemp`。

如果无法建立完整基线，必须标记“无完整基线”，由用户手动选择存档文件，禁止自动判断后直接删除目录。

## 17. 外部备份

外部备份是从游戏盘 `GameSave` 本地存档生成的独立 ZIP 副本，用于游戏盘故障、误删或需要离线保存时的异盘灾备。它不参与普通归档或特殊归档的完成判定，是否配置、生成和校验均不阻止归档。

### 17.1 备份格式

- 使用无密码 ZIP。
- ZIP 内包含存档文件、`manifest.json`、来源游戏盘标识、游戏和版本信息、原始恢复路径及 Hash 清单。
- 不包含游戏解压密码。

### 17.2 手动备份

用户可以对单个游戏执行手动备份。备份完成后进行 ZIP 可读性和文件 Hash 校验。

### 17.3 每日计划备份

- 只支持每天执行，可配置执行时间。
- 通过 Windows 任务计划程序启动独立的无界面备份组件。
- 即使主程序没有运行，也必须能够执行。
- 对所有游戏盘的完整 `GameSave` 目录进行备份。
- 多游戏盘内容合并到一个 ZIP，并按照游戏盘 ID 分目录保存。
- 整体内容指纹没有变化时不重复生成 ZIP。
- 目标磁盘离线时等待磁盘重新上线后补执行。
- 执行完成、部分失败或等待磁盘时弹出 Windows 通知。
- 完整 `GameSave` 计划备份默认保留三个。
- 超过三个时只提醒，不自动删除；用户确认后删除。

示例结构：

```text
GameSaveBackup_20260719_020000.zip
├─ manifest.json
├─ game-disk-{磁盘ID-1}/GameSave/
└─ game-disk-{磁盘ID-2}/GameSave/
```

## 18. 外部备份查看

软件提供只读查看页面，不提供自动恢复：

- 显示备份路径、大小、SHA-256、时间和校验状态。
- 显示 ZIP 内文件树。
- 显示游戏、版本和原始恢复路径。
- 支持打开备份所在目录。
- 支持使用系统程序打开 ZIP。
- 支持复制恢复路径。
- 支持重新校验。
- 根据 `manifest.json` 显示人工恢复步骤。

人工恢复提示至少包括关闭游戏、备份现有目标文件、将 ZIP 中对应内容复制到游戏目录或 Windows 用户目录，以及由用户决定是否覆盖同名文件。

## 19. 原始文件重新定位与删除

### 19.1 重新定位

- 支持选择新文件或目录。
- 支持按文件名、大小、修改时间和 Hash 匹配。
- 检查 ZIP、RAR 格式和分卷完整性。
- 更新路径前要求用户确认替代版本或匹配结果。

### 19.2 删除原始文件

原始压缩包只能通过独立按钮删除，且需要两次人工确认：

1. 第一次显示游戏、版本、路径、文件数量、总体积和无法重新准备的风险。
2. 第二次要求输入版本名称或固定确认文字。
3. 将主压缩文件及相关分卷移入 Windows 回收站。
4. 不支持回收站时终止操作，不得自动永久删除。

删除游戏版本记录时提示是否同时删除原始文件。选择保留时只删除数据库版本记录；选择删除时完成上述二次确认后再删除版本记录。

## 20. 删除与清理规则

| 对象 | 处理方式 | 确认次数 |
|---|---|---:|
| 成功准备任务的 `Games/{游戏ID}/.prepare` | 永久删除 | 0 |
| 成功任务的 `GameSaveTemp` | 永久删除 | 0 |
| 手动清理失败任务临时目录 | 永久删除 | 1 |
| 普通归档游戏目录 | 回收站 | 1 |
| 特殊归档混乱目录 | 回收站 | 1 |
| 原始压缩包和分卷 | 回收站 | 2 |
| 本地存档快照 | 永久删除 | 1 |
| 外部 ZIP 备份 | 永久删除 | 1 |
| 超额历史快照或备份 | 永久删除 | 1 |
| 游戏版本记录 | 数据库删除 | 1 |
| 游戏主记录 | 可勾选同步清理关联数据，输入完整游戏名称 | 1 |

所有无人值守任务不得执行需要人工确认的删除。

删除游戏主记录时提供关联数据复选框，允许用户同步删除版本记录、将可游玩目录移入回收站、永久删除本地存档与外部 ZIP 备份、永久清理任务临时目录、删除系统存档配置，以及解除已保留原始文件关联。未勾选且仍存在的关联数据继续阻止删除；主游戏运行中、存在活动监控或未完成任务时始终禁止删除。同步删除版本记录只删除数据库及版本凭据、基线等从属记录，硬盘上的原始压缩包保持不变；原始压缩包只能通过独立入口完成两次确认后移入回收站。用户必须输入完整游戏名称完成最终确认，确认窗口应以可选择的只读文本单独展示完整名称，支持复制后粘贴到输入框，并明确标示永久删除与回收站操作的区别。

- 文件扫描和来源指纹计算提供取消按钮；准备游玩进度窗口提供取消按钮，并将取消令牌传递到复制、Hash、压缩扫描、解压、游戏目录识别和文件基线步骤。
- 所有使用公共任务进度窗口且允许后台运行的操作统一显示“后台运行”按钮，并启用系统最小化按钮和任务栏入口；系统最小化可从任务栏恢复，点击“后台运行”或关闭按钮则持续隐藏进度窗口且不中断任务，后续阶段更新不得再次抢占前台，用户可在任务中心查看状态。任务实际完成后仍由程序安全关闭进度窗口。
- 用户取消后状态标记为“已取消”，错误时标记为“失败”，临时目录保留；软件启动时将上次仍为“运行中”的任务标记为“已中断”。
- 任务中心对失败、已取消或已中断的准备任务提供“重新发起准备”和“从失败处继续准备”。重新发起前一次确认，将已提交但未落库的 `content` 移入回收站、永久清理 `.prepare` 后执行完整流程；继续准备只复用可验证的 `content` 或能识别有效游戏目录的 `step2`，重新建立基线、恢复存档并补齐游戏状态，无法验证完整性时禁止继续并提示重新发起。
- 存在运行中任务时禁止执行“刷新数据”，避免加载过程把仍在执行的任务错误恢复为已中断并丢失后台任务的最终状态。
- 任务中心支持查看任务详情、打开临时目录、打开关联游戏，以及在一次人工确认后永久清理失败、取消或中断任务的临时目录。
- 临时目录清理前必须验证目标位于已配置游戏盘的旧 `GameTemp`、`GameSaveTemp` 子目录或 `Games/{游戏ID}/.prepare`，禁止删除临时根目录、游戏内容目录、`Games` 或游戏盘根目录。
- 准备游玩直接在 `Games/{游戏ID}/.prepare` 内工作，文件基线成功后同盘移动到 `Games/{游戏ID}/content`，避免大型游戏二次全量复制；失败或取消时不设置可游玩路径。
- 所有阻塞式进度任务以及临时禁用调用窗口的异步任务（包括准备游玩、创建快照、存档扫描、手动或完整备份、密码验证、特殊归档、来源校验和启动前系统存档扫描）完成、取消或失败时，必须先隐藏进度窗口并恢复、激活调用窗口，再显示由调用窗口持有的结果提示；提示关闭并关闭进度窗口后应在界面消息空闲阶段再次恢复调用窗口，调用窗口不得缩小、隐藏或要求用户从任务栏重新切换。
- 成功任务自动永久删除 `.prepare`，无需确认；清理失败不改变游戏已经准备完成的状态，任务中心保留目录供用户后续手动清理。

## 22. 安全与可靠性要求

- 所有危险删除操作遵守确认规则。
- 原始文件和游戏目录优先进入回收站，不自动降级为永久删除。
- 密码使用 DPAPI 加密。
- 防止 ZIP/RAR/7z 路径穿越，禁止解压内容写出目标目录。
- 删除可游玩目录前必须确认本地 `current` 存档清单及文件 Hash 有效；外部 ZIP 备份不作为删除前置条件。
- 复制和备份完成后执行文件清单及 Hash 校验。
- 磁盘空间不足时提前停止。
- 文件被占用时保留数据并显示具体错误。
- 支持 Windows 长路径、中文和特殊字符路径。
- 机械盘默认低并发顺序读取。
- 软件异常退出后能够识别并恢复未完成任务状态。
- 计划任务与主程序同时运行时必须避免数据库并发写入冲突。

## 22.1 安全密码模式与强制加密存储

- 无论安全密码模式是否开启，主数据库始终使用随机 256 位主数据密钥和 AES-256-GCM 加密存储，正式文件为 `data/game-management.secure`，禁止生成包含业务数据的明文临时数据库。
- 安全密码模式关闭时，主数据密钥由当前 Windows 用户的 DPAPI 保护并记录在不含业务数据的 `data/security.json`；后续开启安全密码模式时改由用户密码派生密钥包装同一主数据密钥，无需重新加密全部业务数据。
- 从旧版 `game-management.json` 升级时，必须先写入加密数据库并完成解密回读校验。旧明文数据库在用户明确确认前保留；用户拒绝删除时继续使用加密数据库，并在后续启动持续提示明文泄露风险。
- 加密文件必须校验格式版本和 GCM 认证标签，文件被截断、篡改、密钥不匹配或安全配置损坏时禁止加载，不得静默创建空数据库覆盖原数据。
- 后续阶段继续将游戏图标和运行日志纳入加密范围，并实现密码包装、包装界面、错误次数延迟、自动锁定及锁定状态禁止计划备份。

## 23. 界面规划

应用程序使用独立软件图标：深蓝圆角底板表示稳定的数据管理空间，三层蓝色托盘表示游戏、版本与存档的统一管理，白色游戏手柄表示游戏启动与运行。图标提供高分辨率 PNG、可编辑 SVG 和 Windows 多尺寸 ICO，并嵌入最终单文件 EXE。

界面支持主题切换并将选择持久化到软件数据中，重新启动后自动恢复。1.0.0 提供“经典主题”和“Windows 11 风格”两种主题；Windows 11 风格使用浅色中性背景、圆角按钮、柔和边框、圆角阴影菜单及悬停反馈，并统一应用于主窗口、右键二级菜单和各功能弹窗。主窗口不得使用覆盖全局主题的局部控件样式；顶部标签页、操作按钮、表格外围边框、表头、单元格、行悬停和选中状态均须随主题切换，不能只改变弹窗或菜单。主题切换无需重启，未知或旧数据中的无效主题名称安全回退为经典主题。

- 用户从快捷方式、资源管理器、命令行或计划任务启动时，均应定位到相同的数据目录。
- 软件应检查当前目录的写入权限；不可写时明确提示，不得静默改写到其他目录。
- 软件升级时应保留已有数据库、配置和日志。
- 用户配置的游戏盘、扫描目录和外部备份目录不受默认运行目录限制。
- Windows 计划备份组件必须读取 EXE 所在目录下的同一套配置和数据库。

### 26.4 构建验收要求

- 发布前检查 `build/release/GameManagement.exe` 是否被进程占用；未被占用时直接生成或替换正式 `GameManagement.exe`，不得额外生成 `GameManagement.new.exe`。只有正式 EXE 正在运行或确实无法替换时，才将新产物保存为 `GameManagement.new.exe`，且不得强制结束用户进程。
- 构建产物为单个 EXE 文件。
- 在未安装开发环境和第三方解压软件的干净 Windows 环境中能够启动。
- 将 EXE 复制到新的可写目录后能够首次运行并自动创建数据和日志目录。
- 从其他工作目录启动 EXE 时，数据仍创建在 EXE 所在目录。
- 目录不可写时显示中文错误提示并安全退出。
- EXE 文件移动后能够使用新目录中的数据；不得继续引用旧发布目录的绝对路径。
