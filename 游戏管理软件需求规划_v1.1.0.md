# 游戏管理软件需求规划

## 1. 版本信息

| 项目 | 内容 |
|---|---|
| 规划版本 | 1.1.0 |
| 当前状态 | 规划中 |
| 规划主题 | 游戏目录隐藏与保护 |
| 适用平台 | Windows |
| 当前是否开发 | 否 |
| 当前是否进入测试与发布 | 否 |

本文档仅记录 1.1.0 的候选需求、设计边界和后续实施方向，不表示相关功能已经实现。进入开发前仍需由用户确认最终范围、默认模式、兼容策略和验收标准。

## 2. 规划目标

对已经准备到游戏盘的可游玩目录提供额外保护，降低用户绕过游戏管理软件、直接通过资源管理器识别游戏名称或双击主 EXE 启动游戏的可能性。

1.1.0 规划同时保留以下两种方式，由每个游戏或游戏版本独立选择：

1. 轻量混淆模式。
2. 加密保护目录模式。

两种模式不得同时作用于同一个游戏版本。切换模式前必须先完整解除原保护并通过校验。

## 3. 能力边界

- 轻量混淆只降低目录的可识别性，不构成可靠的访问控制或加密保护。
- 加密保护目录可以在游戏未运行时隐藏真实文件名、目录结构和可执行文件内容，但游戏运行时 Windows 必须能够读取解密后的文件。
- 不承诺抵抗具有管理员权限的用户、内存分析工具、调试器、内核驱动或游戏运行期间的高级访问。
- 不采用进程注入、反调试、驱动级隐藏或修改游戏二进制校验逻辑。
- 不得以保护功能为由破坏游戏文件、存档或原始机械盘压缩包。

## 4. 模式一：轻量混淆模式

### 4.1 状态

当前状态：规划中，未开发。

### 4.2 目标

在不对完整游戏内容加密、不显著增加启动等待和磁盘占用的前提下，隐藏游戏名称和明显的启动文件入口，主要用于防止随手浏览和直接识别。

### 4.3 规划规则

- 游戏根目录继续使用不含中文和游戏名称的游戏 ID。
- 可选设置游戏目录的隐藏、系统属性。
- 原始游戏名称、真实目录名称和启动路径映射仅保存在加密数据库中。
- 主 EXE 只有在兼容性预检查通过后才允许使用随机名称或受控扩展名进行混淆。
- 不批量重命名 DLL、资源包、配置、脚本及游戏运行时可能引用的文件。
- 启动前按映射临时恢复主 EXE 名称或扩展名；游戏结束后再恢复混淆状态。
- 混淆和恢复必须使用同目录原子改名，不额外复制大型文件。
- 每次改名前后记录大小、修改时间和 SHA-256，并在不一致时停止自动处理。
- 游戏运行中禁止修改主 EXE 名称。

### 4.4 兼容性风险

- 部分游戏、启动器、反作弊程序、更新器和插件依赖固定 EXE 名称。
- 修改扩展名不能阻止用户手动恢复，也不能提供真正的加密保护。
- 若兼容性检查无法确认安全，必须禁止自动混淆并建议使用加密保护目录模式。

### 4.5 解除流程

1. 验证混淆映射和文件 Hash。
2. 恢复原始主 EXE 名称或扩展名。
3. 清除隐藏、系统属性。
4. 验证游戏目录和启动文件。
5. 标记为“未保护”。

## 5. 模式二：加密保护目录模式

### 5.1 状态

当前状态：规划中，未开发。

### 5.2 目标

游戏未运行时不在游戏盘保留可直接启动的真实 EXE 和可识别目录结构，将完整可游玩目录转换为经过认证的加密内容块；启动时由软件临时还原运行目录。

### 5.3 规划目录

```text
Games/{游戏ID}/
├─ protected/
│  ├─ content.index.secure
│  ├─ block-000001.secure
│  ├─ block-000002.secure
│  └─ ...
├─ runtime/
│  └─ {随机会话ID}/
└─ protection.state
```

- 加密文件和索引名称不得包含游戏名称、版本名称或真实文件名。
- `content.index.secure` 保存加密后的原始目录结构、文件名、文件属性、大小、Hash 和启动文件相对路径。
- `runtime` 只在解锁或运行期间存在。

### 5.4 加密和密钥

- 每个游戏版本生成独立随机 256 位内容密钥。
- 文件内容按固定大小分块，使用 AES-256-GCM 加密并校验认证标签。
- 内容密钥由现有软件主数据密钥包装，不在日志、命令行或明文临时文件中保存。
- 开启安全密码模式时，必须先解锁软件才能读取内容密钥。
- 忘记安全密码且没有可用恢复方案时，受保护游戏内容同样无法解密。

### 5.5 首次保护流程

1. 检查游戏未运行且不存在准备、归档或备份任务。
2. 计算原目录体积、保护目录空间和安全余量。
3. 建立加密索引并逐块加密。
4. 对全部密文块进行解密回读和 Hash 校验。
5. 只有校验全部通过后才允许请求清理原可游玩目录。
6. 原目录清理必须人工确认并进入 Windows 回收站。
7. 清理失败不影响密文副本，但状态标记为“保护完成、明文待清理”。

### 5.6 启动流程

1. 用户只能通过软件发起已保护游戏的启动。
2. 检查游戏盘空间和保护内容完整性。
3. 解密到 `runtime/{随机会话ID}`，恢复真实目录和主 EXE。
4. 完成文件数量、大小和 Hash 校验后启动主游戏 EXE。
5. 继续使用现有进程监听、存档监控和老板键。
6. 解密、校验和游戏运行期间禁止安全锁定、切换版本及删除保护记录。

### 5.7 退出与重新保护

1. 主游戏进程退出后先完成现有游戏目录与系统目录存档检测。
2. 比较运行目录与加密索引，区分存档变化和游戏本体变化。
3. 游戏本体未变化时清理运行目录。
4. 游戏本体发生变化时询问用户更新密文、忽略变化或暂时保留运行目录。
5. 更新密文必须写入新版本块、回读校验完成后再替换旧块。
6. 运行目录默认进入回收站，不直接永久删除；后续是否允许配置自动永久清理需再次确认。

### 5.8 异常恢复

- 软件启动时检测残留 `runtime` 会话和未完成保护事务。
- 不得自动删除无法判断来源或完整性的目录。
- 提供继续启动、重新保护、验证后清理和打开目录四种人工入口。
- 加密过程中断时保留原明文目录，禁止将半成品标记为已保护。
- 解密过程中断时保留密文内容，残留运行目录按恢复流程处理。

## 6. 统一状态模型

规划新增以下状态：

```text
未保护
轻量混淆中
轻量混淆
正在解除混淆
正在加密保护
已加密保护
保护完成、明文待清理
正在解锁
已解锁待启动
保护模式运行中
正在重新保护
存在残留运行目录
保护失败
等待人工恢复
```

保护状态与现有“未准备、可游玩、运行中、已归档”状态分别保存，不相互覆盖。

## 7. 规划界面

游戏详情和单选右键菜单规划增加“游戏目录保护”二级菜单：

```text
游戏目录保护
├─ 启用轻量混淆
├─ 启用加密保护
├─ 解除当前保护
├─ 查看保护状态
├─ 验证保护内容
├─ 处理残留运行目录
└─ 打开保护目录
```

设置页规划增加：

- 默认保护模式。
- 加密分块大小。
- 运行目录所在游戏盘。
- 游戏退出后的运行目录处理方式。
- 游戏本体发生变化时的处理方式。
- 是否显示高风险轻量混淆选项。

所有清理、解除保护、覆盖密文和丢弃运行目录操作必须提供明确人工确认。

## 8. 与现有功能的关系

- 游戏准备：准备完成后可选择启用一种保护模式。
- 游戏启动：根据保护模式恢复主 EXE或完整运行目录后再启动。
- 游戏归档：先完成存档归档，再处理运行目录；加密游戏本体不作为存档文件。
- 特殊归档：完成特殊归档后才允许建立保护内容。
- 游戏版本：每个版本独立保存保护模式、内容密钥、索引和启动映射。
- 老板键：只隐藏已经运行的窗口和处理音频，不替代目录保护。
- 安全锁定：保护、解锁、重新保护、游戏运行或残留恢复期间禁止锁定。
- 外部备份：现有 GameSave 备份不包含加密游戏本体；保护库备份需作为后续独立需求。
- 原始压缩包：机械盘原始压缩包不参与保护，不因保护游戏目录而删除。

## 9. 规划验收标准

### 9.1 轻量混淆

- 资源管理器中不直接显示游戏名称和明显的主 EXE 名称。
- 软件能够可靠恢复并启动通过兼容性检查的游戏。
- 不兼容游戏拒绝启用，解除混淆后文件 Hash 与保护前一致。
- 文档与界面明确说明其不具备真正的加密安全性。

### 9.2 加密保护目录

- 静止状态不存在可直接启动的真实主 EXE。
- 密文目录不暴露原文件名、目录结构或游戏名称。
- 密文篡改、截断、缺块或密钥错误时禁止启动。
- 解密运行、退出存档、重新保护和异常恢复均不丢失游戏及存档数据。
- 大型游戏展示可取消的进度和明确空间估算。
- 运行期间明确提示临时明文目录仍可被当前 Windows 用户访问。

## 10. 待确认事项

- 1.1.0 是否同时开发两种模式，或优先开发加密保护目录模式。
- 轻量混淆是否允许修改主 EXE 文件名，还是只允许修改扩展名。
- 加密分块大小及是否支持增量更新。
- 运行目录默认进入回收站还是允许自动永久清理。
- 是否提供完整保护库的异盘备份。
- 是否允许对已存在插件、Mod 或外部启动器的游戏启用保护。

以上事项未确认前，1.1.0 保持“规划中”，不得据此直接开始开发。
